Real-Time Transit Data Pipeline & Anomaly Detection

Built a production-grade ETL pipeline ingesting live transit station data into PostgreSQL via Apache Airflow, with Grafana dashboards surfacing real-time anomalies for operational monitoring.

Python PostgreSQL Apache Airflow Grafana ETL Real-Time Pipelines

The Problem

Why This Pipeline Was Built

Transit systems generate continuous streams of sensor and operational data — station throughput, dwell times, ridership counts — that need to be ingested, stored, and made visible to operators in near real-time. Without a structured pipeline, this data sits in raw feeds that no one can easily monitor or act on.

This project built a full data engineering stack from scratch: automated ingestion, a normalized warehouse schema, scheduled orchestration, and live dashboards capable of flagging anomalies as they happen — replicating what a production data team would build for a transit authority.

Methods

What Was Built

1

ETL Pipeline Design

Designed extract-transform-load workflows in Python to pull raw transit station records, validate schema and data types, handle missing values, and write cleaned records to a structured PostgreSQL schema.

2

Apache Airflow Orchestration

Built Directed Acyclic Graphs (DAGs) in Airflow to schedule and monitor pipeline runs, handle retries on failure, and maintain a dependency graph ensuring data arrives in the correct order.

3

PostgreSQL Warehouse Schema

Engineered a normalized relational schema to store ingested transit data efficiently — designed to support both real-time queries for the dashboard and historical analysis over the full dataset.

4

Anomaly Detection Logic

Implemented threshold-based and statistical anomaly flags in SQL — identifying stations with unusually high or low activity relative to their historical baseline during the same time window.

5

Grafana Dashboard

Connected Grafana to the PostgreSQL warehouse and built live panels showing station-level metrics, anomaly alerts, and trend lines — giving operators a single pane of glass for system health.

Outcomes

What the System Delivers

End-to-end automation

Data flows from raw source to live dashboard without manual intervention — Airflow handles scheduling, retries, and logging throughout.

Real-time anomaly visibility

Operators can see flagged stations within minutes of an anomaly occurring, rather than discovering issues in after-the-fact batch reports.

Production-like architecture

The stack mirrors what a real transit authority data team would deploy — PostgreSQL warehouse, orchestrated ETL, and a monitoring layer — built as a full working system, not a prototype.

Maintainable and extensible

DAG-based design makes it straightforward to add new data sources, adjust scheduling, or extend the anomaly detection logic without restructuring the whole pipeline.

Takeaway

This project demonstrates the full data engineering lifecycle — from raw ingestion through orchestration, storage, and live monitoring. Building it end-to-end, rather than in isolated components, gave hands-on experience with the operational realities of production pipelines: handling failures, designing for latency, and making data immediately actionable for non-technical stakeholders.
